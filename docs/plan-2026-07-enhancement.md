# 오늘수영 고도화 기획 (2026-07-09)

> 대상: oneul-swim-frontend + oneul-swim-backend
> 전제: 7/4 완성도 개선 기획의 1~2순위(죽은 UI 제거, 제보 API, Web Push, 서버 배포)는 7/5에 완료됨.
> 이 문서는 그 다음 단계, 즉 "돌아가는 MVP"를 "혼자서도 운영되는 서비스"로 끌어올리는 기획이다.

## 1. 목표와 원칙

- **제품 목표**: 하남 자유수영 정보가 "정확하고, 최신이고, 알아서 알려주는" 상태를 사람 손 최소로 유지한다.
- **포트폴리오 목표**: "기획부터 배포까지 단독 완주"를 넘어 "운영 병목을 시스템으로 해결하는 사람"이라는 서사를 만든다. 각 단계는 블로그 글 1편의 소재가 되도록 설계한다.
- **원칙 (기존 유지)**:
  - 없는 데이터는 날조하지 않는다 (강습 등록일 등).
  - API 장애 시 정적 JSON 폴백으로 항상 동작한다.
  - 과설계 금지: 로그인 없음, 월 0원 인프라 유지.

## 2. 현황 진단 (2026-07-09)

| 영역 | 상태 |
|---|---|
| 핵심 기능 | 시설 4곳 리스트·지도·상세·제보·아침 요약 푸시까지 실동작 |
| 데이터 | 정적 pools.json (2026-06-01 크롤링본). DB 이관·어드민 없음 → **데이터가 썩어도 알 방법이 없음** |
| 품질 | 백엔드 테스트 있음 / 프론트 테스트 0개 (상태 계산·주차 규칙이 무방비) |
| 운영 | 배포는 수동 SSH, DB 백업 없음, 모니터링 없음, 도메인은 sslip.io 임시 |
| 반쪽 기능 | 강습 알림 토글 = localStorage 저장뿐, 실제 발송 없음 |
| 배선 미완 | Vercel 환경변수(NEXT_PUBLIC_VAPID_PUBLIC_KEY) 미등록 → 프로덕션 푸시 미개통 |

핵심 문제 정의: **이 앱의 가치는 "정보의 정확성"인데, 지금 구조는 정보가 틀려져도 감지·수정·전파가 전부 수동(재크롤링+재배포)이다.** 고도화의 중심축은 이 루프의 자동화다.

## 3. 단계별 기획

### P0. 마감 배선: 만든 기능을 진짜로 켜기 (0.5일, 즉시)

이미 구현된 기능의 마지막 스위치. 지원서 라이브 링크가 나가기 전에 반드시 완료.

- Vercel에 `NEXT_PUBLIC_API_URL`, `NEXT_PUBLIC_VAPID_PUBLIC_KEY` 등록 후 재배포 (사용자 직접, Vercel 대시보드)
- 실기기(Android Chrome)에서 푸시 구독 토글 → 다음날 08:00 아침 요약 수신 확인
- 코드 위생: TabBar의 낡은 주석("미구현이라 비활성") 수정

**완료 기준**: 실기기에서 아침 요약 푸시 1회 수신.

### P1. 신뢰 기반: 프론트 테스트 + 데이터 보강 (1~2일)

- **Vitest 도입**: `lib/pools.ts`의 "지금 상태" 계산(open/soon/closed-today/none-today), 둘째·넷째 일요일 등 주차 규칙, 요금 tier 머지에 단위테스트 15~20개. CI에 test 스텝 추가.
  - 이유: 시간·주차 규칙은 엣지케이스 덩어리라 테스트 없이는 데이터 갱신 때마다 회귀 위험. 백엔드(테스트 有)와의 비대칭 해소.
- **좌표·주소 보강**: 카카오 로컬 API 지오코딩 1회성 스크립트로 null 좌표 채우기 → 4곳 전부 지도 핀·길찾기 동작.

**완료 기준**: CI에 프론트 test 스텝 그린, 지도에 4개 핀 전부 표시.

### P2. 운영 자동화: 혼자서도 굴러가는 서비스 (2~3일)

- **CD 파이프라인**: GitHub Actions에서 GHCR 이미지 빌드 → 서버 SSH로 `compose pull && up -d`. main 푸시 = 배포.
- **Prisma migrate 정식 도입**: 현재 `db push` 운영을 마이그레이션 파일 기반으로 전환 (P3의 스키마 확장 전 필수 선행).
- **DB 백업**: 서버 크론으로 일일 `pg_dump` + 보관 7일 롤링.
- **모니터링**: healthchecks.io 무료 플랜으로 `/health`·백업 크론·아침 푸시 크론 감시 (서버 자원 아끼려 Uptime Kuma 자체 호스팅 대신 외부 무료 서비스 권장).
- **도메인**: sslip.io를 DuckDNS 무료 서브도메인으로 교체, Caddy 설정 갱신.

**완료 기준**: main 푸시만으로 배포 완료, 백업 파일 존재, 헬스체크 알림 수신 확인.
**블로그 소재**: "월 0원 인프라 2편: 배포 자동화·백업·모니터링" (긱뉴스 1편 7P의 후속).

### P3. 데이터 파이프라인: 정적 JSON의 부패 문제 해결 (핵심 고도화, 3~5일)

이 프로젝트의 본질적 약점(정보 갱신이 전부 수동)을 시스템으로 푸는 단계. 포트폴리오 서사의 정점.

1. **Pool DB 이관**: Prisma에 Pool·Session·FeeTier 모델 추가, pools.json 시드 스크립트, `GET /pools` API. 프론트는 API 우선 + 실패 시 정적 JSON 폴백(기존 원칙 유지).
2. **신선도 감시 크론**: 주 1회 하남도시공사 페이지를 크롤링해 저장된 스냅샷과 diff → 변경 감지 시 관리자에게 푸시/메일. 자동 반영은 하지 않고 "감지·알림"까지만 (원본 HTML 표가 불안정하므로 사람 검토를 남김).
3. **미니 어드민** (`/admin`, 토큰 인증): 요금·시간표·공지 무배포 수정, 제보 목록에서 PENDING → APPLIED/REJECTED 처리.
4. **제보 루프 완성**: 사용자 제보 → 어드민 검토·데이터 반영 → 제보자의 "내 제보 내역"에 상태 변화 표시. 지금은 제보가 쌓이기만 하는 반쪽 루프다.

**완료 기준**: 재배포 없이 요금 하나를 어드민에서 수정해 앱에 반영, diff 알림 1회 수신, 제보 1건 APPLIED 처리.
**블로그 소재**: "정적 데이터는 썩는다: 4개 수영장 정보를 자동으로 감시하게 만들기".

### P4. 도달 확장 (선택, 각각 독립 실행 가능)

- **TWA 플레이스토어 출시**: assetlinks.json에 실서명 SHA256 반영, Bubblewrap 빌드, 스토어 등록. "스토어에 올린 앱"이라는 한 줄의 가치.
- **강습 알림 실기능화 (재정의)**: 등록일 원천 데이터가 없으므로 D-day 자동 알림은 포기. 대신 어드민이 강습 접수 공지를 입력하면 구독자에게 푸시하는 "접수 소식 알림"으로 스코프 축소. lessons 토글을 localStorage에서 서버 구독으로 승격 (P3 어드민 선행 필요).
- **측정**: Vercel Analytics 또는 GA4 경량 연동. 방문 수 + 핵심 행동(제보 제출, 푸시 구독) 2개 이벤트만. 포폴 라이브 링크의 실사용 근거 확보.

### 백로그 (지금 하지 않는 것, 스코프 방어)

- 아이 수영강습 찾기 (private-pools.json 10곳): 전화·주소·운영시간 미수집이라 데이터 수집 공수가 큼. v3 후보로만 보존.
- 인근 지역(미사 외 구리·성남) 확장: 데이터 파이프라인(P3)이 안정된 뒤에만 의미 있음.
- 로그인·회원: 여전히 과설계. deviceId 익명 모델 유지.
